I build things for the web — backends that hold under pressure, frontends that don't get in the way, and the plumbing in between that nobody sees but everyone depends on. My work lives closest to the Node.js + React ecosystem, though the stack is always secondary to the problem.

I care about three things in this order: does it work correctly, can someone else understand it, and does it hold up at scale. Clean code is the first draft. Production is the real test.

---

### what i work with

**The core stack I reach for on most projects:**

- `Node.js` + `Express` / `NestJS` — REST APIs, service layers, background jobs
- `React` + `Next.js` — SSR, client-side apps, component systems
- `TypeScript` — always, without exception
- `MongoDB` + `MySQL` — document and relational modeling depending on the domain
- `Socket.io` — real-time event systems, live notifications
- `Docker` + `Nginx` — containerized deployments, reverse proxy configuration
- `Redux` / `React Query` / `React Hook Form` — state, server sync, forms

**Where I've built meaningful depth:**

**Payment systems** — Razorpay and Stripe integrations end-to-end: webhook verification, idempotency handling, subscription lifecycle, failure recovery. Not just "add the SDK and call it done."

**API design** — versioned, documented, consistent. I think about the developer consuming the API as a first-class stakeholder, not an afterthought.

**Auth & access control** — JWT flows, refresh token rotation, role-based middleware. Security that doesn't create friction for legitimate users.

**Real-time architecture** — Socket.io for event-driven features. Connection management, room logic, reconnection handling, graceful degradation when sockets aren't available.

### how i work

I tend to write less code than the problem seems to demand, and I think that's usually the right call. Abstraction earns its place — I don't introduce it speculatively.

I'm comfortable owning a feature from database schema to deployed endpoint to React component. I don't disappear into one layer and hand off a bag of assumptions to whoever handles the next one.

Git history should be readable. PRs should be reviewable. Documentation should exist before someone asks for it.
